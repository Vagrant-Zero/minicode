# minicode - 技术方案设计

## 1. 项目概述

### 1.1 背景与目标

在开发复杂项目时，长时间运行的 Agent 任务常面临以下痛点：

| 痛点 | 影响 |
|------|------|
| 上下文丢失 | 长任务中断后无法恢复，需从头开始 |
| 任务进度不可见 | 用户无法感知任务完成情况 |
| 上下文膨胀 | messages 累积导致 token 超限 |
| Agent 能力受限 | 内置工具无法针对特定领域扩展 |
| 外部服务集成困难 | 难以连接数据库、API 等外部服务 |

**minicode** 是一个基于任务驱动开发范式的本地 AI 编程助手，旨在解决上述问题，提供一个适合复杂项目开发、支持渐进式推进、可扩展的本地开发工具。

### 1.2 核心特性

- **任务驱动开发**：任务持久化 + 依赖图管理 + 进度追踪
- **SPEC + PLAN 流程**：需求 → 方案 → 任务的标准化流程
- **上下文管理**：三层压缩策略 + 持久化，告别上下文爆炸
- **统一工具层**：内置工具 + MCP 远程工具 + 插件扩展
- **技能系统**：按需加载领域知识，无需预加载全部
- **后台任务**：异步执行，解决耗时任务阻塞问题

## 3. 核心功能设计

### 3.1 任务驱动开发范式

**背景与问题**：长任务进度不可见，用户无法感知完成情况；任务中断后无法恢复，只能从头开始。

**方法**：设计 TaskManager 实现任务持久化到 .tasks/ 目录，任务支持 blockedBy/blocks 依赖图管理，状态流转（pending → in_progress → completed）。核心原则：**文档优先**，基于文档拆分任务，生成代码，保证项目质量。

**效果**：用户可通过 /tasks 查看进度，任务状态一目了然；中断后可从 .tasks/ 恢复。

**开发流程**：

```
入口：     技术方案
            │
            ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Agent 执行阶段                               │
│                                                                 │
│    1. 任务拆分                                                  │
│       PLAN.md → .tasks/ (任务列表)                             │
│            │                                                     │
│            ▼                                                     │
│    2. 执行任务                                                  │
│       选择未完成任务 → 执行 → 生成代码                           │
│            │                                                     │
│            ▼                                                     │
│    3. 进度追踪                                                 │
│       /tasks 查看 → 继续下一个任务                             │
│            │                                                     │
│            │     ┌─────────────────────────────────────────────┐ │
│            │     │ 不满意？                                   │ │
│            │     ▼                                             │ │
│            │     修改 PLAN.md                                  │ │
│            │     └─────────────────────────────────────────────┘ │
│            │              │                                      │
│            │              │ Agent 重新生成                    │
│            │              ▼                                      │
│            └──────────► 完成                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**核心原则**：
- **入口**：已有 PLAN.md 技术方案
- **Agent 职责**：任务拆分、代码生成、进度管理
- **不满意就迭代** → 修改 PLAN.md → Agent 重新生成 → **不直接改代码**
- **可追溯** → 所有变更记录在 PLAN.md 中

### 3.2 统一工具执行层

**背景与问题**：工具能力有限，难以连接外部服务（如数据库、文件系统、API）。

**方法**：设计 TOOL_HANDLERS 统一分发内置工具（read/write/edit/bash），预留 MCP 扩展接口，启动时从配置文件读取 MCP Server 列表，通过 MCP 协议连接并将远程工具拉平为本地工具。

**效果**：内置工具 + MCP 远程工具统一管理，标准化扩展。

```
┌─────────────────────────────────────────┐
│           统一工具分发层                  │
│      TOOL_HANDLERS 字典统一入口          │
└─────────────────────────────────────────┘
          │           │           │
          ▼           ▼           ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│  内置工具    │ │  MCP 工具   │ │  插件 Hook  │
│ (read/write/│ │(MCP Server) │ │ (生命周期)  │
│  bash/edit) │ │             │ │             │
└─────────────┘ └─────────────┘ └─────────────┘
```

### 3.3 上下文压缩与持久化

**背景与问题**：messages 累积导致 token 超限，上下文膨胀；长任务中断后上下文丢失。

**方法**：设计三层压缩策略——Microcompact 每轮保留最近 3 个 tool_result；Auto-compact 在 token > 100k 时触发 LLM 摘要；Manual 支持用户手动压缩。压缩时保存完整对话到 .transcripts/ 目录。

**效果**：常数级上下文大小，支持中断恢复。

```
Step 1: 检测需要压缩
        ↓
Step 2: 保存完整对话到 .transcripts/timestamp.jsonl
        ↓
Step 3: LLM 生成摘要
        "已完成用户认证模块重构，发现 3 个潜在问题
         下一步：添加 OAuth2 支持"
        ↓
Step 4: 替换 messages
        旧: 200 条消息 (80000 tokens)
        新: 2 条消息 (500 tokens)
```

### 3.4 技能知识系统

**背景与问题**：Agent 能力受限于内置工具，无法针对特定领域扩展。

**方法**：设计 SkillLoader 实现两层注入——Layer 1 描述技能元数据，Layer 2 按需加载完整内容。技能文档放在 skills/ 目录下动态加载。

**效果**：按需加载领域知识，减少 token 消耗。

```
Layer 1: Skill Metadata
┌─────────────────────────────────────────┐
│ name: pdf                               │
│ description: 处理 PDF 文件...           │
└─────────────────────────────────────────┘
              ↓ (按需加载)
Layer 2: Skill Content
┌─────────────────────────────────────────┐
│ # PDF 处理技能                          │
│                                         │
│ ## 读取 PDF                            │
│ pdftotext input.pdf -                  │
│                                         │
│ ## 创建 PDF                             │
│ pandoc input.md -o output.pdf          │
└─────────────────────────────────────────┘
```

### 3.5 插件化扩展机制

**背景与问题**：用户无法自定义工作流或扩展 Agent 能力。

**方法**：设计插件系统，插件放在 plugins/ 目录下动态加载。通过生命周期钩子（before_llm_call、after_tool_execute、on_task_complete 等）在特定阶段注入自定义逻辑。

**效果**：用户可实现自动化测试、代码检查、性能监控、日志记录等扩展。

```
┌─────────────────────────────────────────┐
│            Agent Pipeline                │
├─────────────────────────────────────────┤
│  before_llm_call   → 插件 A, 插件 B   │
│         ↓                               │
│  LLM Call                               │
│         ↓                               │
│  after_tool_execute → 插件 C           │
│         ↓                               │
│  after_llm_response → 插件 D            │
│         ↓                               │
│  on_task_complete   → 插件 E           │
└─────────────────────────────────────────┘
```

### 3.6 后台任务异步执行

**背景与问题**：测试/构建等耗时任务阻塞主循环，LLM 无法继续处理。

**方法**：设计 BackgroundManager 实现 Fire and Forget 模式，后台线程执行耗时命令，通过 drain_notifications 在每次 LLM 调用前抽取结果。

**效果**：耗时任务异步执行，不阻塞主循环。

```python
class BackgroundManager:
    def run(self, command, timeout=120):
        tid = generate_id()
        self.tasks[tid] = {"status": "running"}
        threading.Thread(target=self._exec, args=(tid, command)).start()
        return f"Started: {tid}"

    def drain(self):
        while not self.notifications.empty():
```

### 3.7 上下文隔离与恢复

**背景与问题**：多任务切换时上下文混乱，难以追踪。

**方法**：针对多任务场景，每次开始新任务时读取 .tasks/ 和 .plan.md 重载上下文；任务切换时自动保存当前进度。借鉴 Redis AOF + RDB 思想，持久化任务级执行上下文。

**效果**：任务间上下文隔离，支持中断恢复。

```
任务 A 上下文:
├── .tasks/task_A.json
├── .transcripts/task_A_*.jsonl
└── messages (只包含任务 A 相关)

任务 B 上下文:
├── .tasks/task_B.json
├── .transcripts/task_B_*.jsonl
└── messages (只包含任务 B 相关)
```
